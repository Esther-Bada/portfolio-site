What I Didn't Expect From My First Real Dev Setup

I came into intro week thinking environment setup was a box to tick before the real learning started. I was wrong. Here's what VS Code, three extensions, and Claude Code actually taught me — and why the tools around the code matter as much as the code itself.

Before Any of This

My version of a "development environment" before intro week was Notepad and a browser with too many Stack Overflow tabs open. I'd been dabbling with HTML for a few months — copying tutorials, tweaking things, watching things break without knowing why. I knew what a text editor was. I didn't know what I was missing.

Intro week changed that. And honestly, it wasn't the code itself that shifted the most. It was the tools around the code.

VS Code: More Than a Fancy Notepad

I'd been told to install VS Code before the week started. I did, opened it, stared at the interface for ten minutes, and thought: this is just a text editor with a dark theme. I was wrong, but I didn't know that yet.

The shift happened when I installed my first extension: Live Server. I clicked "Go Live" in the status bar, my browser opened automatically, and every time I saved my HTML file the page refreshed by itself. It sounds trivial. It wasn't. I'd been manually refreshing the browser dozens of times per session for months. That single thirty-second install gave me back a meaningful slice of my working time.

Then came Prettier. My first reaction to auto-formatting was mild annoyance — why is it moving my brackets? My second reaction, after looking at my own code and then at formatted code side by side, was embarrassment. I'd been writing code that was technically valid but visually chaotic. Prettier taught me what consistency actually looks like, and it made reviewing my own work much easier.

GitLens came third, and it was the most conceptually useful. Seeing a tiny annotation next to each line — "you changed this 3 hours ago" — made version control feel visible and real rather than abstract. Git stopped being something I used out of obligation and started being something that showed me the story of my own work.

By the end of the first day, VS Code felt less like a text editor and more like a workspace. That distinction matters more than it sounds.

Claude Code: The Tool I Almost Dismissed

I'll be honest: I came into intro week skeptical about AI coding tools. My assumption was that they were a crutch — something that got you answers without helping you understand them. I was half right, and mostly wrong.

Claude Code isn't a search engine you type questions into. It runs in your terminal, reads your actual files, edits your code directly, and explains what it's doing as it works. The first time I watched it open my stylesheet and make an edit without me pasting in a single line of code, I had to stop and watch it again.

The moment I knew this tool was genuinely different: I had a CSS bug I'd been staring at for nearly an hour. My navbar wasn't positioning correctly. I checked the z-index, the display property, the flex alignment — everything I thought could be wrong. Nothing worked. I described the problem to Claude Code in plain English. It read my stylesheet, identified that I was missing position: relative on the parent container — the one property I hadn't checked — and fixed it. Twenty-five seconds, start to finish.

That wasn't the moment I stopped thinking. That was the moment I started thinking better. Because afterward, I understood why position: relative matters for absolutely positioned children. The fix was instant. The lesson was durable.

What changed my mind about AI tools in a learning context is this: the best use of Claude Code isn't getting it to write code for you. It's using it to unblock yourself faster, so you keep moving forward instead of losing hours to a single frustrating wall. You still have to understand the problem. You still have to know when the output is wrong. You still make the decisions. The tool just compresses the distance between "stuck" and "unstuck."

What the Week Actually Taught Me

I came in thinking setup was something to get through before the real learning started. It turns out the setup is part of the learning.

Choosing and configuring tools forces you to ask: what am I actually trying to do? Installing Live Server forces you to understand the relationship between a source file and the browser. Setting up Git through GitLens makes you think about what a meaningful commit message looks like. Watching Claude Code work in your own files forces you to read code more carefully than you would on your own.

Every tool I added asked something of me in return. Not just time — attention.

My Workflow Now

  1. Open the project folder in VS Code
  2. Start Claude Code in the integrated terminal
  3. Use Live Server to preview changes in real time
  4. Commit with Git using VS Code's source control panel

That's it. Simple, connected, and fast enough that nothing in the environment slows me down before I've even written a line.

If You're Just Starting Out

Don't try to configure everything at once. Install VS Code. Install Live Server. Open your first project. Let that be enough for day one.

Add tools as you feel a specific pain — not because a tutorial told you to. The extensions that stuck for me are the ones I installed because something was annoying me. That's the right signal.

And don't be afraid of AI coding tools. Be skeptical of them — which is different. Use them as a collaborator, not a replacement. Ask it to explain what it did. Read the diff. Push back when something looks wrong. The goal is still to understand what you're building. Good tools just help you get there with less friction.

I'm still early in all of this. But I'm early with a setup that actually works, a workflow I understand, and a clearer sense of what building feels like — not just typing. For week one, that's more than I expected.
